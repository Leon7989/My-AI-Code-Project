# <a name="产品背景"></a>1. 产品背景
全国性高考志愿产品通常需要覆盖全国多个省区市，其产品逻辑天然偏向通用化。

本产品聚焦新疆地区，围绕新疆高考招生计划类型、科类、批次、专项资格、投档规则、历年录取数据以及新疆考生实际需求，建设区域型高考志愿决策辅助产品。

产品不以“简单查询学校”为核心价值，而以：

帮助新疆考生在真实招生规则和可信数据基础上，更高效地完成院校、专业和志愿方案决策。

作为核心目标。

## <a name="一句话定位"></a>2.1 一句话定位

## <a name="核心差异化"></a>2.2 核心差异化
### <a name="差异化一新疆规则深度适配"></a>差异化一：新疆规则深度适配
围绕新疆实际招生体系建立规则引擎，不使用全国通用逻辑直接套用。

2026 年新疆普通高考招生计划类型按照官方规则区分普通类与单列类；推荐系统还需继续考虑考试科目路径、兼报规则、专项资格和批次规则。
### <a name="差异化二中文与维吾尔语双语"></a>差异化二：中文与维吾尔语双语
首页直接提供语言切换：

### <a name="差异化三ai-不是数据源"></a>差异化三：AI 不是数据源
系统禁止让大模型凭内部知识直接决定推荐学校。

核心流程必须为：

用户数据
↓
资格校验
↓
新疆规则引擎
↓
结构化数据库筛选
↓
历史位次分析
↓
风险分层
↓
用户偏好排序
↓
AI解释
↓
生成报告
### <a name="差异化四结果可解释"></a>差异化四：结果可解释
每个推荐结果必须尽可能说明：

- 为什么推荐
- 使用了哪些年份数据
- 用户位次与历史位次关系
- 数据是否完整
- 主要风险是什么
- 是否存在专业限制
- 是否需要查看招生章程

# <a name="产品目标"></a>3. 产品目标
## <a name="用户目标"></a>3.1 用户目标
帮助用户解决：

1. 我的分数和位次大概能上哪些学校？
1. 普通类与单列类规则如何影响我？
1. 我的资格允许填哪些招生计划？
1. 哪些学校适合作为冲刺？
1. 哪些学校相对稳定？
1. 哪些学校适合作为保底？
1. 我的专业偏好是否合理？
1. 是否值得为了学校层次接受冷门专业？
1. 是否值得为了专业接受较低层次学校？
1. 如何组合完整志愿顺序？
## <a name="商业目标"></a>3.2 商业目标
通过免费推荐建立信任，通过完整方案、AI 深度分析、志愿模拟和人工服务完成付费转化。

# <a name="目标用户"></a>4. 目标用户
## <a name="核心用户-a高考考生"></a>4.1 核心用户 A：高考考生
典型特征：

- 关注学校和专业
- 手机使用频率高
- 对 AI 接受度较高
- 容易出现专业选择焦虑
- 对复杂政策理解有限

核心诉求：

- 快速得到候选学校
- 判断冲稳保
- 分析专业前景
- 对比城市和学校
- 获得个性化建议
## <a name="核心用户-b考生家长"></a>4.2 核心用户 B：考生家长
典型特征：

- 更关注录取风险
- 更关注学校层次
- 更关注就业稳定性
- 更关注学费、地域和生活成本
- 对“保底失败”敏感

核心诉求：

- 降低滑档风险
- 看懂政策
- 获得明确解释
- 保存完整报告
- 必要时获得人工协助

# <a name="产品核心原则"></a>5. 产品核心原则
## <a name="数据优先"></a>5.1 数据优先
任何院校推荐必须优先依赖结构化数据。
## <a name="位次优先"></a>5.2 位次优先
推荐逻辑原则上优先使用位次，而不是简单分数比较。
## <a name="规则优先"></a>5.3 规则优先
先判断用户是否具备填报资格，再进行院校推荐。

禁止：

先推荐学校
↓
最后才发现用户不能报
## <a name="ai-不编造"></a>5.4 AI 不编造
AI 不得自主编造：

- 最低分
- 最低位次
- 招生人数
- 招生专业
- 学费
- 专项资格
- 招生政策
- 录取概率
## <a name="来源可追溯"></a>5.5 来源可追溯
关键数据应保存：

- 数据来源
- 来源类型
- 发布机构
- 数据年份
- 抓取时间
- 更新时间
- 审核状态
- 数据版本

## <a name="冲稳保"></a>9.3 冲稳保
### <a name="冲"></a>冲
特点：

- 历史位次存在一定差距
- 有机会
- 风险相对较高
### <a name="稳"></a>稳
特点：

- 用户位次与历史录取区间较匹配
- 综合风险中等或中低
### <a name="保"></a>保
特点：

- 历史位次具有较明显安全空间
- 但不得承诺“保证录取”

## <a name="一次机会的定义"></a>10.2 一次机会的定义
只有满足：

后端成功完成推荐
\+
成功生成结果

相同用户在短时间内提交完全相同参数，不重复扣次。

request\_hash

识别相同请求。

# <a name="隐私与数据保护"></a>18. 隐私与数据保护
## <a name="数据最小化"></a>18.1 数据最小化
仅收集完成推荐所必要的数据。
## <a name="默认不收集"></a>18.2 默认不收集
- 身份证
- 准考证
- 家庭详细地址

除非未来出现具有充分必要性的明确业务。
## <a name="账号能力"></a>18.3 账号能力
用户应拥有：

- 查看个人信息
- 修改个人信息
- 删除部分记录
- 注销账号
## <a name="ai-日志"></a>18.4 AI 日志
尽量避免将直接身份信息放入大模型 Prompt。

建议：

用户ID：内部匿名ID
分数：586
位次：8120
计划类型：普通类

而不是：

姓名：张三
手机号：138...
身份证：...
## <a name="未成年人"></a>18.5 未成年人
产品应考虑部分用户属于未成年人。

涉及未成年人个人信息处理时，应按照适用法律法规建设相应机制。

# <a name="项目风险"></a>28. 项目风险
## <a name="风险-1招生数据质量"></a>风险 1：招生数据质量
这是整个项目最大风险。
## <a name="风险-2规则错误"></a>风险 2：规则错误
新疆规则存在年度变化，禁止将年度政策永久硬编码。
## <a name="风险-3ai-幻觉"></a>风险 3：AI 幻觉
采用数据层和规则层控制。
## <a name="风险-4个人开发者运营能力"></a>风险 4：个人开发者运营能力
需要后台自动化降低人工维护成本。
## <a name="风险-5支付接入"></a>风险 5：支付接入
支付开发前必须根据实际小程序主体、服务类目和商户资质完成接入能力验证。

该事项定义为：

Gate-0

不得假设支付一定可以在当前主体配置下直接上线。
## <a name="风险-6永久会员成本"></a>风险 6：永久会员成本
永久无限 AI 与永久无限人工服务存在长期成本风险。
## <a name="风险-7双语质量"></a>风险 7：双语质量
政策类维吾尔语翻译错误可能产生严重用户信任问题。
